Java’s ForkJoinPool

ForkJoinPool is core concurrency framework provided by Java, and existed from Java 1.7. ForkJoinPool, uses work-stealing to achieve very high throughput. This series of blog posts explores this library.

Intro

In a multi-threaded environment, especially MIMD style computer (i.e most of the processors having multiple cores and threads), work-stealing based scheduling algorithms generally perform quite well. The idea is pretty old1. The core concept being that idle threads pick up the work allocated to other threads, instead of a centralized algorithm trying to dispatch work to threads.

Work-Queues

The inner class Work-Queue implements a dequeue, the standard way2 is to have tasks pushed and popped from one end by the owning thread and popped from other end by stealing thread. The actual implementation uses a circular dequeue design3 which dynamically resizes, and a relaxed condition that a task can be taken multiple times4 and on architectures with very relaxed memory models5 like ARM. The operations are called push, pop and poll (i.e. steal) by the ForkJoinPool.

ForkJoinPool allows max parallelism of 32767. This is because a field called ‘ctrl’ which is 64-bit and holds 4x16bit information including one for number of total workers, allowing 2^15-1 as max parallelism.

An Example: Multiplying Matrices

Let’s start with a simple example. We will use a ForkJoinPool to do recursive matrix multiplication, especially that of a square matrix with power of 2 size. This is a straightforward divide-and-conquer alogrim wiki involving 8 sub-matrix multiplications. This in θ ⁢ ( n 3 ) time complexity (of course, better algorithms exist but this will do for demonstration).

We set the maximum pool size to exactly same as parallelism, (the commonPool tends to keep it around 256+parallelism) and the keepAliveTime is set to minimum (1ms), so threads are terminated quickly. This is generally not a good idea as creating a new thread is quite expensive.

Submitting Tasks to the pool

Any runnable or callable can be submitted to the pool, though there some specialized ForkJoinTask provided. A ForkJoinTask behaves like a lightweight thread. It implement Future interface, so they can be joined. Within the computation of a task, adding new tasks to the pool is done by the .fork() method on the task (or invoke).

  public void compute() {

    long[][] C = new long[size][size];
    if (size <= MAX_ARRAY_SIZE) { // Do not split this work any further 
        // Do the work

    } else {

      addToPendingCount(8); //Since we are going to fork out 8 sub-tasks

      A2 = new MatMulTask(this, A, B, rowA, colA + newSize, rowB + newSize, colB, newSize).fork();
   // ... and other matrices here
   }

    this.result = C;
    tryComplete(); // it is important to call the completer.
  }

Here, for division of tasks we will extend CountedCompleter abstract class. We can use RecursiveTask though this one provides more flexibility. The core method is called compute which is executed inside the pool. During the compute each task will create 8 sub-tasks for the sub-matrix computation by using the fork method. The completer method called onComplete can be overridden by a task. In this example we use to do the construction of resultant sub-matrix after all the sub-tasks are done. A tasks completer (onComplete) would not be called unless all the subtasks are completed.

We do not have to fork into 8, we can just fork into 7 and use the current task to take over the work of 8th task.

Everytime the tasks is forked into sub-task(s), we have tell ForkJoinPool to keep track of how many sub-tasks need to be completed for current task to complete. And that is done by addToPendingCount or setPendingCount.

Every time a task completes, it calls the tryComplete (or its equivalent) which first calls current tasks completer and then calls the parent task’s completer. It keeps on calling until the root task is reached. (A root is the one whose parent is null). This happens before isDone() is called, so trying to join on sub-tasks will actually slow down the process or may even stall.

Do not call .fork() on same task twice!
